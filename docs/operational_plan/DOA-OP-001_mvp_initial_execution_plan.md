# Title

MVP initial execution plan

## 1. Purpose

- Документ описывает план реализации MVP, а не саму реализацию.
- План опирается на **DOA-DEC-001** (baseline architectural decisions).
- Цель: перевести согласованную архитектуру в последовательность исполняемых шагов с проверяемыми результатами.

## 2. Execution Strategy

- Реализация идёт поэтапно.
- Каждый этап даёт проверяемый результат до перехода к следующему.
- Pipeline собирается инкрементально: от данных и retrieval к генерации, оркестрации, сценариям и UI.

## 3. Phases Overview

1. Project bootstrap  
2. Data ingest setup  
3. Retrieval baseline  
4. Generation layer  
5. Orchestration layer  
6. Scenario handling  
7. Web UI  

## 4. Phase 1 — Project bootstrap

**Цель:** создать базовую структуру проекта.

**Включает:**

- Базовый backend skeleton.
- Базовые endpoints (заглушки).
- Подготовку среды выполнения.

**Не включает:**

- Бизнес-логику сценариев.
- Retrieval.
- Подключение LLM / генерацию.

**Результат:** система запускается; есть базовая точка входа для последующих фаз.

## 5. Phase 2 — Data ingest

**Цель:** подготовить синтетический dataset согласно DEC-001.

**Включает:**

- Создание документов (синтетическое наполнение).
- Загрузку данных в систему.
- Подготовку данных к использованию в retrieval.

**Результат:** данные доступны системе для поиска и последующей генерации.

## 6. Phase 3 — Retrieval baseline

**Цель:** реализовать базовый retrieval (vector-based baseline из DEC-001).

**Включает:**

- Поиск по подготовленным данным.
- Top-k выборку релевантных фрагментов.

**Результат:** система возвращает релевантные фрагменты с учётом baseline (без hybrid, reranking, caching).

## 7. Phase 4 — Generation layer

**Цель:** добавить генерацию ответа поверх retrieved context.

**Включает:**

- Формирование ответа на основе запроса пользователя и контекста из retrieval.

**Результат:** система выдаёт связный ответ на запрос при наличии контекста.

## 8. Phase 5 — Orchestration

**Цель:** связать обработку запроса с retrieval и generation в единый поток.

**Включает:**

- Приём и маршрутизацию запроса на уровне оркестрации.
- Последовательный вызов retrieval и generation.

**Результат:** работает end-to-end flow «запрос → контекст → ответ» до добавления полноценного сценарного слоя и UI.

## 9. Phase 6 — Scenario handling

**Цель:** реализовать три сценария MVP: FAQ, equipment selection, equipment overview.

**Включает:**

- Различение и обработку сценария (в т.ч. уточнения для selection при необходимости).
- Согласование поведения с high-level архитектурой и DEC-001.

**Результат:** система различает и обрабатывает три сценария baseline.

## 10. Phase 7 — Web UI

**Цель:** обеспечить обязательный для MVP канал — Web UI (минимальный, demo-oriented).

**Включает:**

- Поле ввода запроса.
- Вывод ответа.
- Историю диалога в объёме, достаточном для демо.

**Результат:** пользователь взаимодействует с системой через UI; демонстрируется полный путь от ввода до ответа.

## 11. Execution Constraints

- Строго следовать **DOA-DEC-001**: demo-first, synthetic dataset, три сценария, Web UI, базовый retrieval.
- Не расширять scope сверх baseline без отдельного решения в decision log.
- Не добавлять advanced retrieval (hybrid, reranking, caching, taxonomy filtering) в рамках этого плана.
- Сохранять demo-first подход: минимально достаточная сложность, проверяемые инкременты.

## 12. Definition of Done

- Реализованы все фазы 1–7.
- Система работает end-to-end через Web UI.
- Поддерживаются три сценария: FAQ, selection, overview.
- Демонстрируется полный pipeline: ingest → retrieval → генерация → оркестрация → сценарии → UI.
- Соответствие границам DEC-001 сохраняется.
