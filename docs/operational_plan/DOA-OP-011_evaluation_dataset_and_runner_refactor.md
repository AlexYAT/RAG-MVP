# Title

Evaluation dataset and runner refactor for type-based metrics

## 1. Goal

Привести evaluation pipeline в соответствие с **DOA-DEC-005**:

- единый dataset с **обязательным** полем `type` у каждой записи;
- **метрики и сводки по каждому `type`** отдельно (стратификация);
- **корректное сравнение keyword vs semantic** в разрезе типов запросов, без опоры только на глобальный average по всему dataset.

## 2. Scope

**Включено:**

- обновление dataset (разметка `type`, покрытие минимального набора типов из DEC);
- обновление dataset loader (поле `type`, валидация);
- обновление evaluation runner (группировка результатов и агрегация по `type`);
- обновление JSON-отчёта (блок `per_type`, сравнение режимов по типам).

**Исключено:**

- изменения retrieval-алгоритмов, индексов и режимов `keyword` / `semantic` (кроме вызова уже существующего API);
- изменения embeddings и vector store;
- UI;
- RAGAS и прочие тяжёлые библиотеки метрик.

## 3. Tasks

### T01 — Dataset update

- Добавить поле **`type`** во все элементы `data/evaluation/eval_dataset.json` (и при наличии CSV-примера — синхронно).
- Обеспечить **наличие `type` у каждой записи**; значения — из набора DEC: `lexical`, `paraphrase`, `noisy`, `scenario`.
- Сохранить **совместимость формата**: те же каналы (JSON `items[]`, CSV с теми же базовыми колонками + колонка `type`).

### T02 — Dataset loader update

- Расширить loader: чтение поля **`type`**, нормализация (trim, lower-case для сравнения с допустимым enum при необходимости).
- **Валидация обязательности `type`:** при отсутствии или пустом `type` — **ошибка загрузки** с понятным сообщением (согласовано с DEC: поле обязательное). Опциональный флаг CLI для legacy-режима **не входит в минимальный scope** OP; при необходимости вынести в follow-up.
- Явно задокументировать в коде/README evaluation: допустимые значения `type` = четыре значения из DEC.

### T03 — Runner refactor

- После прогона по всем строкам dataset **сгруппировать** результаты по **`type`** (без второго полного прогона retrieval: один проход по элементам, два режима keyword/semantic как сейчас).
- Для **каждого `type`** и **каждого режима** (`keyword`, `semantic`) считать отдельно:
  - `avg_correctness`
  - `hit_rate` (по текущим правилам gold для hit@k)
  - `avg_expected_answer_overlap` (или эквивалентное имя поля overlap в отчёте)
  - `hallucination_rate`
- Сохранить **общий summary** по всему dataset (как вспомогательный показатель, в духе DEC-005).

### T04 — Report update

- Добавить в JSON-отчёт блок **`per_type`**, структура вида:

```yaml
per_type:
  lexical:
    keyword: { ... }
    semantic: { ... }
    comparison: { ... }
  paraphrase:
    ...
  noisy:
    ...
  scenario:
    ...
```

(конкретная вложенность может быть `per_type[type][mode]` — зафиксировать при реализации; важно: по каждому типу видны метрики **keyword**, **semantic** и **сравнение**.)

- Сохранить существующий **глобальный `summary`** (whole-dataset).
- Добавить явное **сравнение keyword vs semantic по каждому `type`** (дельты или парные поля внутри `per_type.<type>`).

### T05 — Backward compatibility

- Для dataset **без** поля `type`: **поведение по умолчанию — ошибка загрузки** (см. T02), чтобы не маскировать устаревший формат.
- При необходимости миграции старых файлов: отдельный одноразовый скрипт или ручная разметка — **вне scope** данного OP, если не потребуется явно в execution.

## 4. Risks

- Усложнение runner и отчёта → выше риск регрессий в CLI/JSON schema.
- Ошибки и несогласованность **разметки `type`** в dataset → искажение стратифицированных выводов.
- Неконсистентность строковых значений `type` (опечатки) → падение валидации или «пустые» группы; смягчение: жёсткий enum + явная ошибка при unknown type.

## 5. Output

- Обновлённый evaluation pipeline: dataset с `type`, loader с валидацией, runner со стратификацией, отчёт с `per_type` и сравнением режимов по типам.
- Отчёт с метриками по типам, пригодный для сравнения keyword vs semantic без опоры только на глобальный average.
- **Совместимость с текущим CLI** (`python -m app.evaluation`): те же точки входа; при необходимости добавить только опциональные флаги, не ломая дефолтный прогон на обновлённом dataset.

## 6. Acceptance criteria (execution)

- Прогон на обновлённом dataset завершается успешно.
- В отчёте присутствуют **`per_type`** и **сравнение keyword vs semantic по типам**.
- Отсутствие `type` в элементе dataset приводит к **явной ошибке** при загрузке (согласно T02/T05).
