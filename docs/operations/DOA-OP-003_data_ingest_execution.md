# Title

Data ingest execution

## 1. Context

- Документ — execution-шаг в рамках **DOA-OP-001** (MVP initial execution plan).
- Покрывает **только Phase 2 — Data ingest setup**.
- Опирается на **DOA-DEC-001**: synthetic dataset, demo-first scope, три сценария MVP (FAQ, equipment selection, equipment overview).

## 2. Objective

- Подготовить **синтетический dataset** для MVP, согласованный с baseline решений.
- Сделать данные **доступными системе** на уровне загрузки и чтения (ingest path).
- Заложить **основу для следующего шага** — retrieval baseline — без реализации retrieval в этом шаге.

## 3. Scope

**Включить:**

- Определение минимального состава synthetic dataset под три сценария.
- Создание исходных документов для knowledge base (синтетика, управляемая формулировка).
- Размещение документов в репозитории в согласованной структуре.
- Подготовку ingest path / loading path на минимальном уровне: система может получить содержимое документов программно.
- Возможность загрузки / чтения документов **без** retrieval, embeddings и generation.

**Не включать:**

- Retrieval, vector search, embeddings.
- LLM и generation.
- Classifier, orchestration, сценарную маршрутизацию.
- Web UI.

## 4. Dataset Baseline

**Минимальный состав synthetic dataset:**

- **FAQ:** 3–5 пар вопрос — ответ.
- **Equipment catalog:** 6–9 позиций оборудования (условные устройства).
- **Equipment overview / directions:** краткие описания направлений (областей) номенклатуры.
- **При необходимости:** простые структурированные описания параметров оборудования (для selection/overview), без усложнения формата сверх необходимости для демо.

**Свойства dataset:**

- Синтетический, demo-oriented.
- Управляемый и предсказуемый по содержанию.
- Достаточный для проверки **трёх сценариев** MVP на последующих фазах.

## 5. Tasks

На уровне намерений, без кода и без выбора стека:

1. Определить **структуру** synthetic knowledge base в репозитории (каталоги, соглашения об именовании — настолько, насколько нужно для воспроизводимости).
2. Подготовить **минимальный набор документов** под FAQ, selection и overview.
3. Добавить **механизм загрузки / чтения** этих документов кодом backend (или выделенным модулем в границах проекта) — без индексации и поиска.
4. Обеспечить **проверяемый результат:** по явному сценарию проверки можно убедиться, что документы считаны и представлены системе как вход для будущего retrieval.

## 6. Expected Result

- В проекте **существует** synthetic dataset в согласованном виде.
- Документы **организованы предсказуемо** и пригодны для последующей нарезки / индексации (без выполнения этих шагов здесь).
- Система **умеет загрузить** документы на уровне ingest baseline (чтение из согласованного расположения).
- Следующий шаг retrieval **не блокируется** отсутствием исходных данных.

## 7. Validation

Критерии приёмки этого шага:

- Synthetic dataset **присутствует** в репозитории и покрывает **три сценария** MVP по смыслу (FAQ, selection, overview).
- **Ingest path** работает: документы загружаются / читаются системой в объёме, достаточном для подтверждения (например список путей, количество документов, фрагмент содержимого — способ проверки зафиксировать в implementation snapshot, не в этом документе).
- Шаг **не требует** retrieval, embeddings, LLM или UI для признания выполненным.

## 8. Out of Scope

- Retrieval и индексация, vector database.
- Generation и LLM.
- Scenario routing и orchestration.
- User interface.
- Production-расширения (масштабирование, безопасность данных, непрерывный ingest из внешних систем).
