# Title

Scenario handling execution

## 1. Context

- Документ — execution-шаг в рамках **DOA-OP-001** (MVP initial execution plan).
- Покрывает **только Phase 6 — Scenario handling**.
- Опирается на **DOA-DEC-001**, **DOA-ARCH-001** и **DOA-IMP-005**: orchestration flow уже существует; следующий шаг — **различение и маршрутизация** трёх сценариев MVP (FAQ, equipment selection, equipment overview) **поверх** текущего orchestration baseline.

## 2. Objective

- Добавить **различение сценариев** FAQ / equipment selection / equipment overview.
- **Направлять** пользовательский запрос по соответствующему сценарию поверх существующего orchestration baseline (без замены retrieval/generation внутри шага).
- Подготовить **основу для финального шага** — Web UI — без реализации UI в этом документе.

## 3. Scope

**Включить:**

- **Минимальный** способ определения сценария запроса (правила/эвристики, достаточные для demo-first MVP — без фиксации стека в этом документе).
- **Ветвление** по трём согласованным сценариям MVP и **согласованное** поведение backend для FAQ / selection / overview.
- **Минимальные правила уточнения** для **selection** только если без них ответ **не может** быть осмысленным в рамках демо.
- **Служебную** проверку scenario handling через backend (endpoint или эквивалент) **без** Web UI.

**Не включать:**

- Web UI и любой frontend.
- Сложный dialogue management и долгие сессии уточнений.
- Расширение числа сценариев за пределы baseline.
- Policy layer, кэширование.
- Advanced classifier training / tuning и отдельные ML-пайплайны обучения.
- Новые улучшения retrieval/generation сверх уже зафиксированного baseline.

## 4. Scenario Baseline

Зафиксировать свойства baseline:

- Поддерживаются **только три** сценария: **FAQ**, **equipment selection**, **equipment overview**.
- Scenario handling **надстраивается** над существующим **orchestration flow** (единая точка входа оркестрации остаётся базой; сценарий задаёт ветку поведения до/вокруг вызова оркестрации — детализация в implementation snapshot).
- **Classifier / определение сценария** — **минимально достаточное** для demo-first MVP (прозрачные критерии, воспроизводимость на synthetic dataset).
- **Selection** может требовать **краткое уточнение**, если исходного запроса недостаточно для осмысленного подбора.
- **FAQ** и **overview** должны проходить **без лишней** диалоговой сложности (по возможности — прямой проход к ответу через оркестрацию).
- Результат остаётся **traceable** к retrieval/generation через **orchestration response** (или его расширение полями сценария — без потери связи с `retrieval_trace` / sources).

## 5. Tasks

На уровне намерений, без кода:

1. Определить **минимальные правила** различения трёх сценариев (ключевые формулировки, категории источников, явные маркеры запроса — настолько, насколько нужно для демо).
2. Добавить **backend-слой** scenario handling **поверх** orchestration (вызов оркестрации внутри сценарной ветки или предобработка запроса перед ней — фиксируется в IMP).
3. **Согласовать** response behavior для FAQ / selection / overview (единый каркас ответа с полем сценария, режим уточнения для selection — детали в IMP).
4. Добавить **минимальный механизм уточнения** для selection при нехватке входных данных (один уровень уточнения или явный «запрос недостаточен» — без полноценного state machine).
5. Добавить **служебную проверку** scenario handling **без** UI (явные тестовые запросы на каждый сценарий и на ветку уточнения selection).
6. Обеспечить **проверяемый результат** на текущем **synthetic dataset** и существующем orchestration baseline.

## 6. Expected Result

- Backend **различает** три сценария MVP.
- Запрос **обрабатывается** через соответствующий **scenario path**.
- **Selection** может вернуть **уточнение** при недостаточном вводе (предсказуемый формат ответа).
- Следующий шаг **Web UI** **не блокируется** отсутствием scenario handling: API готов к подключению клиента.

## 7. Validation

Критерии приёмки:

- **FAQ**, **selection** и **overview** **различаются** и обрабатываются **раздельно** (явно видно в ответе или в метаданных сценария).
- На **тестовых запросах** backend возвращает **ожидаемый тип** сценария / режима ответа.
- **Selection-path** поддерживает **минимальное уточнение** при необходимости (воспроизводимый сценарий в IMP).
- Scenario handling **можно проверить отдельно** от Web UI.
- Для признания шага выполненным **не требуется** frontend-реализация.

## 8. Out of Scope

- Web UI и расширенный product dialogue.
- Дополнительные сценарии вне трёх baseline.
- Advanced classifier / ML improvements.
- Policy / security hardening и production-оптимизация.
