# Title

LLM baseline integration into RAG pipeline

## 1. Overview

Цель данного плана — **выполнить внедрение** зафиксированного **LLM baseline** в RAG MVP в соответствии с **DOA-ARCH-005** (слои embedding capability и LLM-based generation, контекст, минимальный orchestration) и **DOA-DEC-006** (единый provider **OpenAI**, embeddings и generation через API, без multi-provider и без abstraction layer на этом этапе).

План описывает **порядок работ и критерии приёмки**, **без** новых архитектурных решений и **без** расширения scope сверх перечисленного ниже.

## 2. Execution Scope

**Включено:**

- интеграция **embeddings** (вызов API, получение векторов для запроса и согласованное использование в retrieval);
- интеграция **LLM generation** (grounded промпт, использование собранного контекста);
- обновление **сквозного pipeline** ответа пользователю.

**Исключено:**

- оптимизации (латентность, стоимость, батчинг);
- новые подсистемы (reranking, кэш, сложный orchestration engine);
- расширение **evaluation** (новые метрики, новые датасеты, интеграция в CI).

## 3. Tasks

### T1 — Embedding integration

- Подключить вызов **OpenAI embeddings API** согласно **DOA-DEC-006**.
- Реализовать **единую функцию/модуль** получения embedding для переданного текста (ошибки API → явная обработка, без silent fallback на «пустой» вектор).
- **Ingestion** не перестраивать радикально: при необходимости только минимальные правки путей данных, необходимые для подачи текста в embedding и в индекс.

**Проверка:** unit- или ручной вызов функции embedding возвращает осмысленный вектор фиксированной размерности для тестовой строки.

### T2 — Retrieval update

- Обеспечить использование **semantic сигнала** (embedding запроса и сопоставление с хранимыми представлениями фрагментов) в **retrieval layer** в духе **DOA-ARCH-005**.
- **Сохранить** существующий **keyword retrieval** как доступный режим или fallback согласно уже принятой логике проекта (без введения **hybrid** сложности: не комбинировать несколько скорингов в один обязательный путь).

**Проверка:** на одном и том же запросе видно различие или согласованное поведение режимов; semantic путь использует embedding.

### T3 — Generation integration

- Подключить **LLM generation** через OpenAI API (chat/completions-семейство — как принято в реализации, без смены DEC).
- Реализовать **grounded** промпт: в промпт попадает только контекст из **context builder** (retrieval hits), явное требование не выдумывать факты вне контекста.
- Использовать существующий контур **context builder** (сборка текста и источников из hits); при необходимости — минимальное расширение полей, без новой подсистемы.

**Проверка:** ответ содержит ссылку на источники / метаданные, согласованные с retrieval.

### T4 — Pipeline integration

- Собрать end-to-end цепочку: **query → retrieval → context → generation → response**.
- Обновить точку входа генерации ответа (например, `pipeline` / `answer` слой приложения) так, чтобы **по умолчанию** использовался LLM-generation с grounded-контекстом, в рамках MVP-политики ошибок (нет ответа → понятное сообщение).

**Проверка:** один вызов публичного API или CLI-скрипта проходит полный путь без ручной подстановки промежуточных данных.

### T5 — Smoke validation

- Прогнать **несколько** заранее выбранных запросов (FAQ-тип, equipment-тип).
- Убедиться:
  - в логах или трассировке виден **embedding** шаг для semantic retrieval;
  - итоговый текст пришёл из **LLM**;
  - в ответе или метаданных присутствуют **sources** из retrieval.

**Проверка:** короткий чеклист результатов зафиксирован в IMP или в комментарии к PR (вне scope данного OP — на усмотрение исполнителя).

## 4. Execution Order

**T1 → T2 → T3 → T4 → T5**

Параллельное выполнение не требуется; T4 опирается на завершение T1–T3.

## 5. Acceptance Criteria

- **Embeddings** используются в **retrieval** (semantic сигнал присутствует в рабочем пути).
- **LLM** используется для **generation** итогового ответа пользователю.
- **Pipeline** работает **end-to-end** от запроса до ответа с контекстом из retrieval.
- Поведение остаётся **MVP-уровня**: понятные ошибки при отсутствии ключа или сбое API, без новых обязательных подсистем из §6.

## 6. Non-Goals

- **Reranking** и сложное ранжирование.
- **Caching** ответов и embeddings.
- **Multi-provider** и смена DEC без нового документа.
- **Оптимизация стоимости** токенов.
- **Расширение evaluation** и обязательная перенастройка отчётов под новый generation (кроме минимальной совместимости, если потребуется отдельным микро-шагом вне scope этого OP).
