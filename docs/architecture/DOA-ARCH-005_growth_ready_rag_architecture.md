# Title

Growth-ready RAG architecture with minimal LLM baseline

## 1. Overview

Этот документ **детализирует архитектуру** следующего lifecycle RAG MVP в соответствии с **DOA-IDEA-005**: продуктово-ориентированный контур, **scalable-by-design** и **minimal-by-scope**, с **обязательным минимальным LLM baseline** — семантическим сигналом через **embeddings** в retrieval и **LLM-based generation** с **grounding** в извлечённом контексте.

Цель раздела — зафиксировать **архитектурный каркас** (границы компонентов, поток данных, точки расширения и ограничения), **без** выбора конкретных технологий, хранилищ, моделей или поставщиков. Реализация и операционные планы выносятся за рамки ARCH.

## 2. High-Level Architecture

Система рассматривается как набор **логических слоёв** со стабильными интерфейсами между ними:

| Компонент | Роль в каркасе |
|-----------|----------------|
| **Ingestion layer** | Приём и нормализация знаний (документы, фрагменты), подготовка артефактов для индексации. |
| **Embedding layer** | **Capability**: преобразование текста (запросов и/или корпуса) в **семантические представления** для retrieval. Обязателен как концепт для lifecycle IDEA-005; конкретный механизм вычисления не фиксируется здесь. |
| **Retrieval layer** | Поиск релевантных фрагментов по запросу. **Сейчас** в продуктовом контуре допускается опора на **keyword-oriented** сигнал; **semantic retrieval** описывается как **целевое направление** того же слоя (тот же контракт «запрос → упорядоченный список фрагментов с метаданными»). |
| **Context builder** | Сборка **контекстного пакета** для генерации: отбор, усечение, упорядочивание фрагментов и служебных полей (источники, идентификаторы) без смены смысла retrieval-результата. |
| **Generation layer** | **LLM-based** формирование **итогового ответа** пользователю при **обязательном grounding**: выход опирается на переданный контекст, а не на произвольное доменное дополнение. |
| **Orchestration layer** | **Минимальный**: последовательность шагов «запрос → retrieval → контекст → генерация → ответ» и точки для подключения политик (без описания сложного движка). |
| **Evaluation compatibility** | Внешний по отношению к runtime-ответу контур измерений: те же **входы/выходы** pipeline (запрос, извлечённые источники, ответ) должны оставаться **наблюдаемыми** для существующего evaluation **без** требования расширять сам evaluation в этом ARCH. |

Принцип **growth-ready**: новые возможности подключаются через **явные extension points** (§5), не ломая базовый поток §4.

## 3. Component Responsibilities

### Ingestion layer

- **Назначение:** обеспечить воспроизводимую подготовку знаний к индексации и последующему поиску.
- **Входы:** сырые документы домена (в границах политики проекта).
- **Выходы:** структурированные единицы знания (фрагменты + метаданные), пригодные для индекса keyword и/или семантического канала.
- **Ограничения:** не дублирует retrieval и не генерирует пользовательский ответ.

### Embedding layer (capability)

- **Назначение:** дать **семантический сигнал** для retrieval (и при необходимости для сопоставления запроса с корпусом).
- **Входы:** текст (фрагмент корпуса и/или пользовательский запрос — по политике pipeline).
- **Выходы:** вектор или иное согласованное представление, потребляемое retrieval-слоем.
- **Ограничения:** не подменяет generation; не является обязательным местом бизнес-логики сценариев.

### Retrieval layer

- **Назначение:** вернуть **ранжированный** набор фрагментов и метаданных, достаточный для context builder.
- **Входы:** нормализованный запрос; опционально — семантическое представление от embedding layer.
- **Выходы:** список «hit» с текстом, оценкой релевантности, ссылкой на источник в корпусе.
- **Ограничения:** не формирует натурально-языковой ответ пользователю; keyword и semantic рассматриваются как **режимы или этапы** одного слоя с общим контрактом выхода.

### Context builder

- **Назначение:** превратить сырой результат retrieval в **ограниченный по объёму** контекст для LLM с сохранением трассируемости источников.
- **Входы:** список hit-ов; параметры политики (лимиты, порядок).
- **Выходы:** контекстный блок (текст + структурированные источники) для generation layer.
- **Ограничения:** не добавляет факты извне retrieval-результата.

### Generation layer (LLM-based)

- **Назначение:** сформулировать **ответ пользователю** на языке интерфейса, **опираясь** на переданный контекст (**grounded**).
- **Входы:** пользовательский запрос; контекст от context builder; политика стиля/безопасности (на уровне требований, без привязки к провайдеру).
- **Выходы:** текст ответа; структурированные ссылки на использованные источники (для объяснимости и evaluation).
- **Ограничения:** не является источником истины о фактах вне переданного контекста в рамках заявленного grounded-поведения.

### Orchestration layer (минимальный)

- **Назначение:** задать **порядок** вызовов слоёв и единую точку входа для сценария «один запрос — один ответ».
- **Входы:** пользовательский запрос; конфигурация режимов (например, retrieval profile) на уровне абстракции.
- **Выходы:** финальный ответ и метаданные прогона (достаточные для логов и evaluation).
- **Ограничения:** не превращается в сложный rule engine в рамках описываемого каркаса.

### Evaluation compatibility

- **Назначение:** сохранить **наблюдаемость** тех же артефактов, на которых строится текущий evaluation (в т.ч. стратификация по типам запросов на уровне датасета).
- **Входы/выходы:** согласованы с контрактом «запрос — hits — ответ» без расширения набора метрик в этом документе.
- **Ограничения:** evaluation не является частью runtime-критического пути ответа пользователю.

## 4. Data Flow

Пошаговый поток **по умолчанию** (минимальный сценарий):

1. **Query** — пользовательский запрос попадает в orchestration.
2. **Retrieval** — retrieval layer возвращает релевантные фрагменты; при использовании семантического канала **embedding layer** участвует **до или внутри** шага поиска (как согласовано в реализации), обеспечивая **семантический сигнал**.
3. **Context building** — context builder формирует ограниченный контекст и набор источников.
4. **Generation** — **generation layer (LLM)** формирует ответ, **grounded** в переданном контексте.
5. **Response** — orchestration возвращает ответ и метаданные (в т.ч. для evaluation и объяснимости).

**Где используется embedding:** на пути «запрос и/или корпус → семантическое представление → **retrieval**», т.е. embedding **обслуживает retrieval**, а не подменяет генерацию.

**Где используется LLM generation:** на шаге **Generation** — единственный слой, отвечающий за **натурально-языковую** сборку итогового ответа при соблюдении grounding.

## 5. Extension Points (Design for Extension)

Ниже — **допустимые** направления эволюции каркаса; **в текущем lifecycle они не обязаны быть реализованы** и не расширяют обязательный scope IDEA-005 на уровне поставки:

- **Pre-validation / query classification** — вставка логического шага **до** retrieval (или до выбора профиля retrieval) для фильтрации, маршрутизации или выбора политики; контракт: тот же выход «нормализованный запрос + опциональные подсказки для downstream».
- **Scenario-based prompt building** — расширение **context builder** и/или **generation** за счёт шаблонов и сценарных политик без смены факта «ответ из LLM grounded в извлечённом контексте».
- **Расширение orchestration** — дополнительные ветки (например, уточняющий вопрос, отказ в ответе) при сохранении **явных** границ между слоями.

Все перечисленные пункты — **точки роста**, а не обязательные узлы текущего каркаса.

## 6. Constraints

- **MVP-first:** минимальный сквозной путь «запрос → ответ» важнее полноты вспомогательных подсистем.
- **No production complexity:** отказ от требований уровня промышленной эксплуатации в этом ARCH (масштабирование, SRE, формальная security-программа — вне документа).
- **No provider lock-in:** архитектура не закрепляет конкретного поставщика или проприетарного стека; слои общаются через **контракты** (данные и роли), а не через имена продуктов.
- **Compatibility с текущим evaluation:** сохраняется возможность подачи на evaluation тех же наблюдаемых артефактов (запрос, источники, ответ), **без** обязательства менять сами метрики или pipeline оценки в этом цикле.

## 7. Non-Goals (Architecture level)

- **Multi-agent systems** и распределённая координация нескольких автономных агентов.
- **Сложные orchestration engines** (графовые workflow-движки как обязательная часть каркаса).
- **Provider-specific архитектура** (жёсткая привязка интерфейсов к одному вендору или одному семейству API).
- **Обязательная привязка к конкретному классу хранилища** (например, обязательный «векторный СУБД» как архитектурный must): retrieval остаётся **абстракцией** с наблюдаемым контрактом выхода.
