# Title

RAG MVP quality and efficiency improvements

## 1. Context

Система RAG MVP уже реализована и работает в end-to-end режиме (см. `DOA-IMP-014`). Текущий контур включает semantic + keyword retrieval, grounded LLM generation и единый pipeline ответа. По итогам внешнего аудита выявлены практические проблемы эксплуатации и качества, которые требуют отдельного улучшительного цикла без изменения базовой продуктовой рамки.

## 2. Goal

Повысить качество эксплуатации, эффективность, соответствие документации и безопасность поведения системы, сохранив совместимость с текущим MVP-пайплайном.

## 3. Scope

### A. Documentation alignment

- Актуализировать `README` под фактический pipeline.
- Отразить актуальные env-переменные, зависимости и режимы retrieval.

### B. Repository hygiene

- Исключить runtime-артефакты (`.chroma/`) из репозитория и рабочего процесса коммитов.

### C. Index efficiency

- Устранить полное пересоздание embeddings/index при каждом старте.
- Перейти к более эффективной стратегии обновления/переиспользования индекса.

### D. Error handling safety

- Предотвратить утечку внутренних ошибок в пользовательский ответ.
- Разделить user-facing сообщения и внутреннюю диагностическую информацию.

## 4. Out of scope

- Не вводится reranking.
- Не вводится caching (кроме индекса при необходимости).
- Не вводится multi-provider.
- Не вводится agent orchestration.
- Не изменяется UI.
- Не добавляется новая функциональность RAG сверх улучшений качества/эксплуатации.

## 5. Success criteria

- `README` соответствует фактическому состоянию системы.
- Runtime-артефакты не попадают в git.
- Индексация не пересоздаётся полностью при каждом запуске.
- Пользователь не видит внутренние ошибки системы.
- Поведение системы остаётся совместимым с текущим MVP.

## 6. Risks

- Усложнение стратегии индекса может повлиять на стабильность.
- Возможна деградация retrieval при некорректной оптимизации индекса.
- Избыточное masking ошибок может скрыть полезную диагностику для разработчика.

## 7. Notes

- Изменения носят улучшительный характер.
- Базовый pipeline не должен ломаться.
