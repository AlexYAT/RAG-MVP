# Title

Evaluation dataset strategy with query types for RAG evaluation

## 1. Context

- Evaluation pipeline реализован и зафиксирован как baseline (**DOA-IMP-011**).
- Базовые метрики получены и могут использоваться для регрессий.
- Текущий синтетический dataset в силе формулировок **не репрезентативен** для оценки semantic retrieval и для честного сравнения стратегий retrieval.

## 2. Problem

- Dataset **смещён** в сторону lexical / keyword-дружественных запросов.
- **Отсутствует** структурирование по типам запросов.
- При таком смещении **невозможно корректно сравнивать** keyword vs semantic retrieval: агрегированные средние **искажают выводы** о преимуществах режимов.

## 3. Options considered

**A** — расширение dataset без структуры (только больше вопросов).

**B** — разделение на **несколько** dataset-файлов (например, по режимам или по сценариям).

**C** — **единый** dataset с явным полем **`type`** у каждого элемента.

## 4. Decision

Принят вариант **C**: **единый dataset** с **обязательным** полем **`type`** для каждого элемента.

Обоснование кратко: один артефакт проще версионировать и прогонять end-to-end; поле `type` даёт стратификацию без дублирования пайплайна и без ручного сопоставления нескольких файлов.

## 5. Dataset structure

Каждый элемент dataset **должен** содержать:

- `id` — стабильный идентификатор строки.
- `question` — текст запроса.
- `type` — **обязательное** поле типа запроса (значение из согласованного набора, см. §6).
- `expected_answer` — опционально; для overlap / judge при наличии эталона.
- `expected_sources` — опционально; для hit@k и трассировки источников.

## 6. Supported query types

Минимальный набор типов (расширение только через явное обновление DEC или наследника):

- **`lexical`** — точные или близкие к документу формулировки (высокая lexical overlap).
- **`paraphrase`** — перефразированные запросы без дословного совпадения с корпусом.
- **`noisy`** — неполные, опечатки, лишние слова, фрагментарный ввод.
- **`scenario`** — пользовательские сценарии («нужно для отделения X», «какой монитор для Y») в естественной формулировке.

## 7. Evaluation implications

- Метрики retrieval и ответа **считаются отдельно по каждому `type`** (стратифицированные сводки).
- Сравнение **keyword vs semantic** **проводится в разрезе типов** (и при необходимости по подмножествам с gold-разметкой).
- **Агрегированный average по всему dataset** используется **только как вспомогательный** показатель, а не как единственный критерий сравнения режимов.

## 8. Consequences

**Положительные:**

- Более **корректное** сравнение retrieval-стратегий и снижение риска ложных выводов из-за keyword bias.
- Возможность **выявить зоны**, где semantic retrieval даёт выигрыш (например, paraphrase / scenario).
- Опора на **data-driven** последующие решения (DEC/IMP) с явной стратификацией.

**Отрицательные:**

- **Усложнение** подготовки и сопровождения dataset (нужна дисциплина разметки `type`).
- Потребуется **обновление evaluation runner** и отчёта: группировка и вывод метрик по `type` (вне scope данного DEC; фиксируется как следствие решения).

## 9. Non-goals

- **Не** вводится RAGAS и не расширяется до тяжёлых библиотек метрик.
- **Не** добавляется UI для разметки или просмотра evaluation.
- **Не** усложняется базовая метрика сверх необходимого для стратификации по `type` (без новых сложных составных индексов в рамках этого решения).
