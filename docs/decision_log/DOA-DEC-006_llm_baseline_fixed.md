# Title

LLM baseline fixed for embeddings and generation

## 1. Context

В **DOA-ARCH-005** зафиксирован архитектурный каркас следующего lifecycle: **embedding layer** как capability для семантического сигнала в retrieval и **generation layer** как **LLM-based** формирование ответа с **grounding** в извлечённом контексте.

На уровне **уже принятого** проектного решения необходимо **зафиксировать в decision log** конкретный **LLM baseline**: один поставщик, использование LLM для embeddings и для generation, отказ от multi-provider и от отдельного abstraction layer на данном этапе, с учётом доступности **OpenAI API key** для разработки и демо.

## 2. Decision

Зафиксировано следующее (как **принятое** состояние, без повторного сбора вариантов):

- Используется **единый provider** для всех LLM-операций в рамках baseline.
- В качестве **текущего** provider выбран **OpenAI**.
- **Embeddings** реализуются через **LLM-based** (API embeddings) подход в retrieval pipeline.
- **Generation** реализуется через **LLM** (chat/completions-семейство API) с политикой grounded-ответа из контекста.
- **Multi-provider** поддержка **не требуется** на текущем этапе.
- **Abstraction layer** под нескольких поставщиков **не вводится** на данном этапе.
- Для работы baseline предполагается **наличие OpenAI API key** в среде выполнения (разработка / демо).

## 3. Rationale

- **Минимальная сложность** реализации и сопровождения одного интеграционного контура.
- **Соответствие DOA-IDEA-005**: обязательное использование LLM для семантического представления и для генерации ответа.
- **Наличие API ключа** снижает риск блокировки интеграции на старте цикла.
- **Ускорение перехода к реализации** без проектирования provider-agnostic прослойки до появления реальной потребности.

## 4. Alternatives Considered

- **Multi-provider** — **отклонено** как избыточное усложнение для текущего MVP-этапа.
- **Локальные модели** — **отклонено**: для текущего baseline не требуется.
- **Отказ от LLM** в одном из двух обязательных слоёв (embeddings или generation) — **недопустим**: противоречит принятому **DOA-IDEA-005**.

## 5. Consequences

**Плюсы:**

- Быстрый запуск интеграции и проверки end-to-end.
- Простая архитектура с одной точкой учёта учётных данных и квот.

**Минусы:**

- **Зависимость от одного provider** (OpenAI) — принимается как допустимый компромисс на текущем этапе; смена поставщика потребует отдельного решения и работ, не входящих в этот DEC.

## 6. Non-Goals

- Выбор и одновременная поддержка **нескольких моделей** как обязательного требования baseline.
- Введение **abstraction layer** под нескольких поставщиков в этом решении.
- **Оптимизация стоимости** (бюджетирование токенов, маршрутизация по цене) — вне scope данного DEC.
- Требование **provider-agnostic** архитектуры на текущем этапе.
