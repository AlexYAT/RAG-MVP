# Title

Environment Variables Strategy for MVP

## Context

- В проекте уже используется `.env` файл в `backend`.
- Автоматическая загрузка `.env` в runtime отсутствует.
- `python-dotenv` не используется.
- Переменные (включая `OPENAI_API_KEY`) доступны процессу только при явной установке в окружении ОС.

## Problem

- Разработка становится менее удобной: перед каждым запуском нужно вручную подготавливать окружение.
- Возрастает риск ошибок запуска (забытая переменная, запуск в «чистом» терминале).
- Нет единого и явно зафиксированного подхода для MVP-цикла.

## Options

### Option 1 — System environment only

Использовать только переменные окружения ОС.

### Option 2 — dotenv only

Использовать `python-dotenv` и `.env` как основной источник переменных.

### Option 3 — Hybrid (dotenv + env)

Использовать `dotenv` в development, а системные переменные окружения — как fallback и production-механизм.

## Decision

Выбран **Option 3 — Hybrid (dotenv + env)**.

- В development допускается и рекомендуется загрузка `.env` через `dotenv`.
- В production и в CI приоритет остаётся за переменными окружения системы.
- При наличии переменных в окружении ОС они продолжают рассматриваться как валидный источник конфигурации.

## Why

- Улучшается удобство разработки и локального запуска.
- Сохраняется совместимость с production-практикой (system env).
- Подход остаётся гибким для MVP и не усложняет архитектуру.

## Consequences

**Плюсы:**

- Проще локальный запуск.
- Меньше ошибок, связанных с пропущенными переменными.

**Минусы:**

- Появляется дополнительная зависимость (`python-dotenv`).

## Non-goals

- Не внедряются секрет-менеджеры.
- Не внедряются vault/infra-решения.
